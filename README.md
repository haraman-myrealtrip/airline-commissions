### 2. 데이터 흐름 (핵심)

이 대시보드는 **"HTML 하드코딩 = 초기값/폴백", "구글시트 = 실제 화면에 뜨는 값"** 2단 구조입니다.

```
페이지 로드
  │
  ├─ 1) loadEmbeddedData()   → index.html 안에 하드코딩된 배열로 화면 초기화
  │        airlines[]  (커미션 요약 표/카드)
  │        historyData[] (상세 이력 표)
  │
  └─ 2) loadSheetsData()     → 구글시트를 읽어 위 값을 "덮어씀"
           대시보드_커미션 탭  → airlines[]의 각 항목을 코드 기준으로 갱신
           대시보드_히스토리 탭 → historyData[]를 통째로 교체
```

**중요**: 그래서 HTML만 수정하고 시트를 안 바꾸면, 로드 시 시트값이 이겨서 **라이브 화면은 안 바뀝니다.** 항목을 실제로 반영하려면 **HTML(코드 기준값)과 구글시트(화면 값)를 둘 다 동일하게** 맞춰야 합니다.

## 사이트 문제점 및 TODO

### [1] 원인 — "왜 토큰 만료가 반복되는지"

현재 관리자 저장 기능은 구글 로그인 시 받은 **1시간짜리 임시 권한(토큰)** 으로만 동작합니다.
1시간이 지나면 권한이 만료되는데, 자동 재발급이 없어 저장 버튼을 눌러도 아무 일도 안 일어나는 현상이 발생합니다.
참고로 **시트 → 사이트 반영 기능은 토큰과 무관**하다고 보이며, 페이지 새로고침 및 구글 캐시 때문에 반영이 늦어 보일 수 있습니다.

### [2] 단기 대응 (차주)

저장하기 직전에 토큰이 만료됐는지 확인하고, 필요하면 자동으로 재발급 후 저장하도록 하는 로직 추가가 필요하다고 보여집니다.
대부분의 경우 사용자는 재로그인 없이 계속 사용할 수 있습니다.

### [3] 장기 대응 (지속적인 사용을 위한 개선 방안)

브라우저에서 직접 구글 시트를 수정하는 구조를 없애고, **Google Apps Script를 중간 서버처럼 사용**하는 방식으로 전환을 검토해볼 수 있을 것 같습니다.
이렇게 변경하면 토큰 만료 개념 자체가 사라져, 장시간 사용해도 저장이 끊기지 않을 것으로 기대됩니다.

## [추가 발견] 데이터 구조 문제 — "하드코딩이 실제 base, 시트는 오버레이"

> 2026-06-30 코드 분석 중 발견. **가장 근본적인 구조 이슈**로, QA 노트의 "구글시트와 대시보드 구조가 일치하는지 모르겠다"의 원인입니다.

### 현상: 전체 데이터가 3곳에 하드코딩되어 있음

| 위치 | 내용 | 상태 |
|------|------|------|
| `index.html`의 `loadEmbeddedData()` → `airlines[]` | 커미션 **81개** 전부 하드코딩 | 배포본 |
| `index.html`의 `loadEmbeddedData()` → `historyData[]` | 히스토리 **25개** 전부 하드코딩 | 배포본 |
| `commission.html`의 `airlines`/`history` | **또 다른 복사본**(포맷도 다름) | 링크 안 된 고아 파일, 방치 |

### 런타임 동작

```
loadEmbeddedData()   ← 하드코딩 81+25개를 "먼저" 세팅 (항상 실행)
        ↓
loadSheetsData()     ← 그 위에 구글시트를 "덮어씀(overlay)"
   · 커미션(대시보드_커미션 탭): 시트에 존재하는 코드만 필드별로 덮어씀 = 패치 방식
   · 히스토리(대시보드_히스토리 탭): 탭에 데이터가 있으면 → 통째로 교체
                                     탭이 없거나 비어있으면 → 하드코딩 25개 그대로 유지
```

즉 **진짜 기준값(source of truth)은 HTML 하드코딩이고, 구글시트는 그 위에 덮는 얇은 오버레이**입니다. 이로 인해:

- **`대시보드_히스토리` 탭이 없다면 → 히스토리는 사실상 100% 하드코딩으로 동작**하며 시트 연동이 무의미합니다. (실제로 이 탭의 존재 여부 확인 필요 — 아래 체크리스트)
- **커미션은 시트가 기존 81개 코드의 "값"만 덮어씀** → 시트로는 **항공사 추가/삭제가 불가능**하고, 신규 항공사는 반드시 HTML을 수정해야 생깁니다.
- 데이터 소스가 HTML·시트로 이원화되어 **불일치·드리프트(drift)** 가 구조적으로 발생합니다.

### ✅ 복귀 시 먼저 확인해야 할 것

1. 구글시트 하단 탭 바(왼쪽 아래 ▤ *모든 시트 보기*)에서 아래 탭의 **실제 존재 여부 + 데이터 유무** 확인
   - `대시보드_커미션`
   - `대시보드_히스토리`   ← **이 탭이 안 보인다는 제보 있음. 우선 확인 필요**
   - `변경이력`
2. 사람이 관리하는 상세 탭(예: 오버컴 상세)과 위 "앱 전용 탭"이 **서로 다른 탭**임을 인지 (앱은 위 3개 이름의 탭만 읽음)

## [추가 발견] 개선 로드맵 — "구글시트를 단일 소스로"

현재까지의 개선(실시간 폴링)은 "오버레이를 자주 읽기"까지이며, 근본 해결은 **하드코딩 = base 구조 자체를 걷어내는 것**입니다.

**목표: 구글시트를 단일 소스(Single Source of Truth)로, HTML은 시트를 렌더만 하는 얇은 뷰로 전환**

- 앱 시작 시 시트에서 전량 로드 / 하드코딩은 **오프라인 폴백 용도로만** 축소
- 항공사 추가·삭제·히스토리 관리를 **전부 시트에서만** → HTML 수정 불필요
- `commission.html` 죽은 복사본 정리

**제안 순서**
1. **시트 탭 상태 확인** (위 체크리스트)
2. 탭이 있고 데이터가 차 있으면 → "시트 = 단일 소스" 리팩터 진행
3. 탭이 비어있으면 → 하드코딩 81+25개를 시트로 1회 이관(seed)한 뒤 전환

> ⚠️ 전제: 이관/전환 전에 반드시 `대시보드_커미션`/`대시보드_히스토리` 탭에 전체 데이터가 존재해야 함. 안 그러면 전환 순간 데이터가 비어버림.

### 그 외 구조 리스크 (참고)

- **히스토리 저장은 append가 아니라 탭 전체를 클라이언트 메모리로 덮어씀** → 두 명이 거의 동시에 다른 항공사를 저장하면 한쪽 변경이 유실될 수 있음(동시성 위험). 진짜 이력 보존은 `변경이력`(append-only) 탭에만 의존.
- **첨부 이미지는 구글 드라이브 "링크 공개" 파일** → 업로더 계정/파일 삭제 시 대시보드 이미지가 깨짐(시트엔 URL만 저장).

## 작업 브랜치: `update-by-iamhope/0630`

담당자 부재 기간 중 iamhope가 별도 브랜치에서 선반영한 개선(리뷰 후 머지 검토용, `main` 미반영):

**실시간 연동**
- 로그인 시 1회만 읽던 구조 → **60초 자동 폴링 + 탭 재활성화 시 즉시 동기화**
- **수동 `↻ 새로고침` 버튼** 및 마지막 동기화 시각 표시
- gviz **캐시버스터(`_cb`)** 로 시트 반영 지연 완화 / 관리자 편집 중·백그라운드 탭에서는 폴링 skip

**중복 로그인 해결**
- 세션(email/name)을 `localStorage`에 저장 → **새로고침 시 재로그인 없이 열람** (읽기는 공개 gviz라 토큰 불필요한 점을 활용)
- 관리자 Sheets 쓰기 스코프는 **'저장' 시에만** 요청 → **로그인 시 이중 OAuth 팝업 제거**
- 시트 생성 보장 로직은 관리자 토큰 획득 시점으로 이동(idempotent)

> ⚠️ **미검증**: OAuth 로그인이 필요한 실동작(팝업 1회화·저장)은 헤드리스 환경에서 확인 불가. **실제 계정으로 검증 필요.** JS 문법 체크는 통과.
